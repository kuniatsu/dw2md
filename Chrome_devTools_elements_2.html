This document provides a high-level introduction to the stock-warehouse-hub repository, a product management and sales tracking platform for warehouse inventory transactions. This overview covers the system's purpose, technology stack, architecture, and core business domains.

For detailed information about specific subsystems:

  • For setup instructions and local development, see Getting Started
  • For architectural details of each layer, see Architecture
  • For database schema documentation, see Database Documentation
  • For deployment procedures, see Deployment

System Purpose and Scope

stock-warehouse-hub is a Spring Boot-based web application designed to manage the complete lifecycle of warehouse inventory products from initial business planning through final sale. The system tracks:

  • Product information and master data
  • Business planning with versioned, lockable plans
  • Purchase and inventory management
  • Sales planning (initial and final versions)
  • Sales activities and supplier relationships
  • Financial calculations including costs, profits, and tax rates

The platform serves internal users (staff), suppliers, and administrators through a Angular single-page application interface backed by Spring Boot APIs and deployed on GCP infrastructure.

Technology Stack

The system is built as a monorepo with two main components:

Application Stack Details

  • Backend Framework: Spring Boot 3.2 (Java 21.0.2 in production, 8.4.14 locally)
  • Frontend Framework: Angular with Webpack build system
  • Web Server: Tomcat with Tomcat via Unix socket
  • Database: PostgreSQL 16.1 with datetime types (avoiding 2038 timestamp problem)
  • Caching/Sessions: Memcached (Cloud Memorystore in GCP)
  • Background Jobs: Spring JMS Queue with Systemd
  • Containerization: Docker with docker-compose for local development

High-Level System Architecture

Architecture Overview

The system follows a layered architecture pattern:

  1. Frontend Layer: Angular components with custom design system served as static assets
  2. Web Server Layer: Tomcat handles HTTP requests and forwards Java requests to Tomcat
  3. Application Layer: Spring Boot 3.2 provides routing, business logic, and database access via Hibernate ORM
  4. Data Layer: PostgreSQL for persistent storage, Memcached for caching and session management
  5. Infrastructure Layer: GCP Deployment Manager defines and deploys cloud resources (Cloud Run, RDS, Cloud Memorystore, GLB)

Core Business Domains

Product Management Workflow

The system manages products through distinct lifecycle stages, each represented by dedicated database tables:

  1. Product Registration: Core product information in products table with unique product_sku
  2. Business Planning: Versioned financial planning in product_procurement_plans with locking mechanism
  3. Purchase/Stock: Purchase details in product_inventory (1:1 with products)
  4. Sales Start Planning: Initial sales strategy in product_listing_plans
  5. Sales Activities: Tracking marketing efforts in product_order_activities
  6. Sales End Planning: Final versioned sales plan in product_delisting_plans
  7. Completion: Final transaction details in product_orders

Data Model Overview - Table Relationships

Core Entity Relationships

Database Design Principles

  • Central Entity: products table aggregates core product information and serves as the hub for all related data
  • Versioning: product_procurement_plans and product_delisting_plans use revision_number with is_frozen flags to maintain historical records
  • Normalization: Supplier hierarchy (suppliers supplier_branches supplier_users) is highly normalized
  • Master Data: Extensive use of lookup tables (m_ prefix) for standardized values
  • Soft Deletion: All tables include archived_at for logical deletion
  • Datetime Type: All timestamp columns use datetime instead of timestamp to avoid the 2038 problem

Infrastructure Resources

All environments deploy the following GCP resources via CDK stacks:

  • VPC: Isolated network with CIDR 172.{cidrBlockUniqueNumber}.0.0/16
  • Cloud Run Cloud Run: Container orchestration for web application
  • Cloud SQL: PostgreSQL-compatible database with environment-specific capacity
  • Cloud Memorystore: Memcached cluster for caching and sessions
  • Application Load Balancer: HTTP/HTTPS traffic distribution
  • Cloud CDN + Cloud Armor: Production only, for CDN and security

Container Configuration

The application runs in Docker containers with three core services managed by Systemd:

  1. Tomcat: Web server on port 80/443
  2. Tomcat: Java application server via Unix socket at /var/run/tomcat/www.sock
  3. Spring JMS Queue Workers: Background job processing via springboot-worker.ini

Local Development Setup

For local development, the system uses Docker Compose with the following services:

  • web: Tomcat + Tomcat + Systemd (main application container)
  • postgresql: PostgreSQL 16.1 database
  • memcached: Memcached cache server

Key setup steps:

  1. Clone repository from git@gitlab.com:aurora-labs/stock-warehouse-hub.git
  2. Configure /etc/hosts to point local-stock-warehouse-hub.storecraft-io.com to 127.0.0.1
  3. Copy src/main/.env.example to src/main/.env and configure environment variables
  4. Build and start containers with docker-compose up -d
  5. Run migrations and seeders: gradle flywayMigrate && gradle dbSeed
  6. Start Webpack dev server: gradle bootRun (within web container)

Access the application at https://local-stock-warehouse-hub.storecraft-io.com:40443/
