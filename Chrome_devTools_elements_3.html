Purpose and Scope

This document provides a high-level overview of the gymflow-api system, a comprehensive membercare management platform built as a terraform microservices architecture on GCP. This overview introduces the system's primary domains, deployment structure, and architectural patterns. For detailed information about specific subsystems, refer to:

  • Architecture and deployment details: Architecture
  • Gym staff-facing features: Gym Domain
  • Subscriber-facing features: Member Domain
  • Company management: Administration Domain
  • Photo service: Gallery Domain
  • Authentication and authorization: Authentication & Authorization

System Description

The gymflow-api is a multi-tenant membercare management system that serves three primary user types: operation company administrators, gym staff, and subscribers. The system manages the complete lifecycle of membercare operations including member enrollment, daily activity tracking, meal management, financial billing, subscriber communication, and media gallery services.

The platform is organized into four major business domains:

DomainPurposePrimary UsersCloud Endpoints
GymDaily membercare operations and managementGym staffMain Cloud Endpoints
HomeSubscriber access to member information and communicationSubscribersHome Cloud Endpoints
OperationMulti-gym company administrationOperation company adminsOperation Cloud Endpoints
GalleryMedia gallery service with e-commerceStaff (upload), Subscribers (purchase)Gallery Cloud Endpoints + Gallery Home Cloud Endpoints

Additionally, a Hardware domain handles Smart device integration for QR code scanners and sensor devices.

High-Level Architecture

The system implements a modular microservices architecture where each domain is decomposed into multiple Cloud Function service stacks to avoid Terraform's 500-resource limit and enable independent deployment cycles.

Core Domain Entities

The system's data model centers around a multi-tenant hierarchy where operation companies manage multiple gym schools, which enroll memberren and employ staff.

Key Entity Models:

  • franchise_company: Represented by internal/model/pg/adminCompanyModel.go
  • gym_facility: Represented by internal/model/pg/gymFacilityModel.go
  • member: Managed via internal/handler/gym/member/service.go and internal/model/pg/memberModel.go
  • user_facilities: Staff management in internal/handler/gym/staff/service.go41-72
  • subscriber_user: Subscriber management in internal/handler/member/subscriber/service.go
  • gym_class: Class management in internal/handler/gym/facility/service.go22-37

Sources: internal/handler/admin/facility/service.go1-218 internal/handler/gym/staff/service.go1-316 internal/handler/gym/facility/service.go1-169

Request Processing Architecture

The system implements a three-layer request processing pattern consistently across all domains:

Example flow for gym school creation (createAdminFacilityService):

  1. Handler: internal/handler/admin/facility/handler.go receives Cloud Endpoints event
  2. Request Interface: internal/handler/admin/facility/request.go87-136 validates and transforms input using @cloudendpoint() decorator
  3. Service: internal/handler/admin/facility/service.go68-128 implements business logic:
    • Validates user authorization via checkUserBelongFranchiseCompany
    • Creates GymFacilityEntity via gymFacilityModel.createGymFacility
    • Creates initial admin user via userFacilitiesModel.createUserFacilities
    • Seeds initial form data via Step Functions
  4. Response: Returns formatted OperationGymResponse via internal/handler/admin/facility/response.go

Sources: internal/handler/admin/facility/service.go68-128 internal/handler/admin/facility/request.go87-136 internal/handler/gym/staff/service.go41-72

Authorization Model

The system implements a three-tier hierarchical authorization structure:

Authorization Functions:

  • checkUserBelongFranchiseCompany: internal/handler/admin/facility/utils.go - Validates operation company membership
  • checkUserBelongFacility: internal/handler/gym/staff/service.go41-72 - Validates gym membership and IP restrictions
  • checkUserHasManagerAuth: Used in internal/handler/gym/staff/service.go192 - Validates admin privileges
  • IP Whitelisting: internal/handler/gym/staff/service.go61-70 - Checks geoWhitelistModel when isGeoRestricted

Sources: internal/handler/gym/staff/service.go41-72 internal/handler/admin/facility/service.go42

Technology Stack

Runtime Environment:

  • Go 1.22 on GCP Cloud Functions (AMD64 architecture)
  • Go for type safety
  • Terraform 1.7 for deployment

Data Storage:

  • PostgreSQL Cloud SQL: Primary relational database (config/\\{stage\\}/constant.yml - pg_host, pg_port)
  • Firestore: High-velocity transactional data (config/\\{stage\\}/constant.yml - firestore_prefix)
  • Cloud Storage: Three-tier storage (private, public, operation buckets)

External Integrations:

  • Firebase Auth: OpenID Connect authentication (config/\\{stage\\}/constant.yml - open_id_* variables)
  • Stripe Payment: Card processing (config/\\{stage\\}/constant.yml - stripe_* variables)
  • Firebase Cloud Messaging: WebSocket real-time events (config/\\{stage\\}/constant.yml - firebase-messaging_* variables)
  • Beams: Mobile push notifications (config/\\{stage\\}/constant.yml - beams_* variables)
  • SendGrid: Transactional email (config/\\{stage\\}/constant.yml - ses_home_email, ses_gym_email)
  • GCP Step Functions: Asynchronous workflows (config/\\{stage\\}/constant.yml - stepfunctionsBaseArn)

Testing:

  • Go testing for unit and integration tests
  • Comprehensive test suite: test/request/test.go1-2543 with 100+ test cases
  • Service-level tests: test/service/test.go1-796

Sources: terraform/modules/app.tf16-18 terraform/modules/app1.tf16-18 terraform/modules/app4.tf48-102

Deployment Configuration

Each service stack is configured with stage-specific environment variables loaded from config/{stage}/vars.yaml:

Key Configuration Files:

  • Database: pg_host, pg_port, pg_user, pg_password, pg_database
  • Storage: gcp_bucket_name, gcp_public_bucket_name, gcp_bucket_name_operation
  • API Keys: app_api_key, dev_api_key, app_api_key_member, app_api_key_admin
  • VPC: vpc_private_subnet1, vpc_private_subnet2, vpc_private_security_group

Deployment Strategy:

  • Local Development: Stage local - All services in monolithic stack (terraform/modules/app.tf115)
  • Development: Stage dev - Modular independent stacks (terraform/modules/app1.tf109)
  • Production: Separate deployment buckets per stack to isolate artifacts
