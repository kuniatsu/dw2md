Purpose and Scope

The vetclinic-core system is a Django-based web application for managing online doctor hospital registrations, staff accounts, and related administrative functions. The platform enables hospital administrators to register their medical facilities, configure online consultation services, manage staff credentials, and handle multimedia content including promotional videos and hospital imagery.

This document provides a high-level overview of the system architecture, core components, technology stack, and primary functional domains. For detailed information about specific subsystems, see:

  • Security implementation details: Security Architecture
  • Database table relationships: Database Schema
  • Hospital registration workflow: Hospital Registration Workflow
  • Staff management features: Personnel Management System
  • Authorization and access control: Authorization System

Key Architectural Patterns:

  1. Multi-Stage Data Processing: Hospital registration follows a three-stage pattern (new.py confirm.py complete.py) with session-based state management
  2. Database Abstraction: All database operations route through wrapper functions in common.py (db_query, db_create, db_execute)
  3. Temporary-to-Permanent File Pipeline: Uploads land in /images/tmp/, move to permanent storage only after confirmation
  4. Role-Based Authorization: Access control implemented via check_permissions() in lib-hospital-auth.py

Security Model Overview

The system implements a five-layer security architecture with multiple independent authentication and authorization mechanisms:

Layer 1: Transport Security

  • HTTPS Enforcement: All HTTP requests redirected to HTTPS via URL dispatcher middleware.py46-48

Layer 2: Network Access Control

  • IP Whitelisting: Admin paths (/wp-login.py) restricted to 12 whitelisted IP addresses middleware.py4-20
    • Aurora: 10.20.30.40, 10.20.30.41
    • TechPartner: 10 additional addresses
  • User-Agent Bypass: vetclinic_agent user agent bypasses authentication middleware.py2

Layer 3: HTTP Authentication

  • Token Authentication: General access requires digest auth with username ondocadmin middleware.py24-28
  • Session Authentication: Admin directory requires separate basic auth admin/middleware.py referenced

Layer 4: Application Authorization

  • Role-Based Access Control: Implemented via check_permissions() function in lib-hospital-auth.py
    • Returns is_approved_action flag and user role
    • Role check: veterinary_staff role has restricted permissions
  • Staff Type Differentiation: role_type field in hospital_staff table determines access levels

Layer 5: Data Security

  • Password Hashing: make_password() with PBKDF2_SHA256 (bcrypt) admin/hospital/complete.py476
  • SQL Injection Protection: Database wrapper functions (db_query, db_create, db_execute) in common.py
  • Session Management: PHP sessions with duplicate submission prevention admin/hospital/complete.py1-21

File Processing Pipeline

The system implements a three-stage file upload pipeline to ensure data integrity:

StageLocationPurposeTriggered By
1. Upload/images/tmp/Temporary storage during form submissionimage_upload.py::handle_upload()
2. Confirmation/images/tmp/ (retained)User reviews data with hidden form fields preserving filenamesconfirm.py
3. Commit/images/ or /images/thumbnail/Permanent storage after final submissioncomplete.py::move_uploaded_file()
4. CleanupN/ADelete all /images/tmp/ contentscomplete.py:965-967

// From admin/hospital/complete.py:6-15
function move_uploaded_file($filename, $dir="") {
    $tmp_dir = $_SERVER['DOCUMENT_ROOT']."/images/tmp/";
    $uploaddir = $_SERVER['DOCUMENT_ROOT']."/images/".$dir;
    if (rename($tmp_dir.$filename, $uploaddir.$filename)) {
        return true;
    } else {
        echo "";
        return false;
    }
}

Key System Characteristics

Multi-Tenancy Model:

  • Each hospital has a unique hospital_id that serves as the primary organizational key
  • Staff accounts (hospital_staff) can be linked to multiple hospitals via hospital_link table
  • Flexible hospital type system (category_type) enables tiered feature access

Session-Based Workflow Protection:

  • Registration completion uses $_SESSION["hospital_regist"] to prevent duplicate submissions admin/hospital/complete.py20-21
  • Session check pattern: if session variable set, display error; otherwise proceed with insertion

Database Abstraction:

  • All database operations route through wrapper functions: db_connect(), db_query(), db_create(), db_execute()
  • Logging enabled via $query_log variable pointing to dated log files
  • Dual database support: production (pet_hospital_db) and demo (pet_hospital_db_demo1)

Soft Deletion Pattern:

  • Records not physically deleted; instead status=9 flag marks deletion
  • delete_date timestamp preserved for audit trail
  • Queries filter deleted records with WHERE status != 9 clauses
